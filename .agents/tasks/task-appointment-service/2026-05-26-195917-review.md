# Terminal Appointment Service review pass (v2)

The fix commit addresses the five critical issues from v1: status transitions are now enforced via an `ALLOWED_TRANSITIONS` state machine, the frontend shows container number fields for all four required transaction types, the BulkUpload component uses selectors instead of hardcoded values, external service return values are now checked before creation, and appointment number generation uses 6 hex digits with retry-on-collision. The remaining three acknowledged items (auth stub, non-atomic bulk, no body size limit) are accepted as intentional for this phase.

**Watch for:** CANCELLED status reachable via the update endpoint bypassing cancel-specific side effects (confirmed), terminal ID mismatch between AppointmentForm and BulkUpload components (confirmed), `Math.random()` still used for appointment numbers despite retry logic (possible â€” adequate for current scale but not cryptographically sound for production).

**Verdict**: NEEDS_CHANGES

## High-level view

The state machine fix introduces a side-channel: setting `appointmentStatus: "CANCELLED"` via the update endpoint bypasses the dedicated cancel flow's slot release, notification dispatch, cancellation reason recording, and timestamp. The state machine permits the transition (PENDING/CONFIRMED -> CANCELLED) without routing through the compensating logic.

The two frontend forms now use different terminal ID schemes. AppointmentForm hardcodes `"TRAPAC-LA"` and `"TRAPAC-OAK"` (the terminal `code` field), while BulkUpload hardcodes `"term-la-001"` and `"term-oak-001"` (the test-seeded `terminalId` primary key). Since the Appointment model's `terminalId` column is a foreign key referencing `Terminal.terminalId` (a UUID in production), only one scheme can be correct per environment, and neither will work with auto-generated UUIDs from the production seed.

## CANCELLED reachable via update endpoint without cancel semantics

The state machine permits PENDING->CANCELLED and CONFIRMED->CANCELLED as valid transitions. When a caller sends `PUT /api/appointments/:id` with `{ appointmentStatus: "CANCELLED" }`, the update path executes:

```typescript
if (data.appointmentStatus) {
  const currentStatus = existing.appointmentStatus;
  const allowedNext = ALLOWED_TRANSITIONS[currentStatus];
  if (!allowedNext || !allowedNext.includes(data.appointmentStatus)) {
    throw new ServiceError(...);
  }
  updateData.appointmentStatus = data.appointmentStatus;
}
```

This sets the status to CANCELLED but skips everything the cancel endpoint does: setting `cancellationReason`, recording `cancellationTimestamp`, calling `slotService.releaseSlot`, sending TOS cancellation notification, dispatching user notification, and writing the audit log with a reason field.

The result is an appointment that appears cancelled but has no reason, no timestamp, an unreleased slot, and no downstream system notification. The fix is either to remove CANCELLED from the ALLOWED_TRANSITIONS targets entirely (making the dedicated cancel endpoint the only path) or to detect a transition to CANCELLED in the update handler and delegate to the cancel logic.

## Terminal ID divergence between frontend components

AppointmentForm defines:
```typescript
const TERMINALS = [
  { id: "TRAPAC-LA", name: "TraPac Los Angeles" },
  { id: "TRAPAC-OAK", name: "TraPac Oakland" },
];
```

BulkUpload defines:
```typescript
const TERMINALS = [
  { id: "term-la-001", name: "TraPac Los Angeles" },
  { id: "term-oak-001", name: "TraPac Oakland" },
];
```

The `Appointment.terminalId` column is a FK to `Terminal.terminalId`, which is `@id @default(uuid())`. The test setup manually assigns `"term-la-001"` as the `terminalId`, but the production seed uses `upsert` without specifying `terminalId`, letting Prisma generate a UUID. `"TRAPAC-LA"` is the `code` column, not the primary key.

AppointmentForm will fail with a FK violation in both environments (test and production) because `"TRAPAC-LA"` is never a valid `terminalId`. BulkUpload works against the test database but will fail in production where UUIDs are auto-generated.
