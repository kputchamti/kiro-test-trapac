# TraPac Terminal Appointment Service (v3)

Full-stack appointment management system for TraPac terminals: Express/Prisma/SQLite backend with state-machine-enforced lifecycle, Zod validation, audit trail, and bulk creation with per-item error semantics. Next.js 14 frontend with transaction-type-aware forms, client-side validation on both single and bulk paths, and aligned terminal IDs across components. All three issues from v2 have been addressed: CANCELLED removed from state machine transitions, terminal IDs unified to test-fixture values, and BulkUpload now validates before submission.

**Watch for:** Production seed generates UUID terminal IDs while frontend hardcodes test-fixture IDs — will require a terminal lookup API or seed alignment before any non-dev deployment (confirmed). This is a deployment-time concern, not a code correctness issue for the current development phase.

**Verdict**: APPROVED

## High-level view

The state machine now funnels all cancellation through the dedicated endpoint, which handles slot release, TOS notification, reason recording, and audit. The update endpoint rejects any transition to CANCELLED via both the state machine map and the Zod enum constraint, providing defense in depth.

Both AppointmentForm and BulkUpload reference `"term-la-001"` / `"term-oak-001"` as terminal IDs, matching the test database's seeded values. The production seed uses `@default(uuid())` without specifying a `terminalId`, so the hardcoded values won't resolve against a freshly-seeded production database — a terminal lookup API is the obvious pre-production fix.

BulkUpload validates terminal, trucking company, transaction type, reference number, and time fields client-side before submission. It does not validate container number based on transaction type (unlike AppointmentForm), so a PICK_IMPORT row without a container number passes client-side but is rejected by the backend with a per-row error. Minor UX friction, not a data integrity gap.
