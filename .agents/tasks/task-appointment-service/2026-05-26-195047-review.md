# Terminal Appointment Service greenfield implementation

A full-stack appointment management system for the TraPac terminal: Express backend with Prisma/SQLite, Next.js 14 frontend, Zod validation, transaction-type-specific business rules, audit trail, and per-item error handling for bulk creation. The approach is service-oriented with external integrations stubbed behind interfaces. The architecture is sound for a first pass, but the auth story is entirely absent, the appointment number generator will produce collisions at scale, and the update endpoint permits arbitrary status transitions that bypass the lifecycle rules enforced on cancel.

**Watch for:** No authentication or authorization (confirmed), appointment number collision risk via `Math.random()` (confirmed), unconstrained status transitions on the update path (confirmed), bulk endpoint runs sequentially without transactional boundaries (confirmed), frontend hardcodes `terminalId` and `truckingCompanyId` in bulk upload (confirmed).

## High-level view

The backend is structured as a single `AppointmentService` class that owns the lifecycle: create, update, cancel, bulk-create, list, and audit retrieval. External concerns (slot management, TOS notifications, validation) are abstracted behind interfaces with stub implementations that always succeed, so the current code has no error paths for external service failure — return values from validation and slot checks are never inspected.

The auth middleware extracts `x-user-id` from a header and falls back to "system" when absent. There is no token verification, no role-based access control, and no tenant isolation. Every endpoint is open to any caller who can reach the network.

The status lifecycle is enforced asymmetrically: cancel has a strict allow-list of non-cancellable statuses, but the update endpoint allows any caller to set `appointmentStatus` to any string value, including transitions that should be illegal (e.g., CANCELLED -> CONFIRMED). The Zod schema on the update path validates it as `z.string().optional()` with no enum constraint.

The frontend validation for transaction-type-specific fields diverges from the backend: the form shows container number inputs only for PICK_IMPORT and DROP_EXPORT, but the backend also requires them for DROP_EMPTY and PICK_EMPTY. The BulkUpload component hardcodes `terminalId: "TRAPAC-LA"` and `truckingCompanyId: "default-company"`, which will fail against the backend's foreign key constraints unless those exact seed values exist.

<summary>Issues (8)</summary>

1. **No authentication or authorization** — The auth middleware accepts any `x-user-id` header with no verification. Add JWT/session validation and role-based guards before any production deployment.
2. **Appointment number collision** — `generateAppointmentNumber()` uses `Math.random()` for a 4-hex-digit suffix (65,536 values per day). Use a database sequence or UUID-based approach to guarantee uniqueness.
3. **Unconstrained status transitions via update** — `PUT /api/appointments/:id` accepts any `appointmentStatus` string. Enforce a state machine (PENDING->CONFIRMED->CHECKED_IN->COMPLETED, etc.) with explicit allowed transitions.
4. **Bulk creation is non-atomic and sequential** — Each item in a bulk request is created independently. A partial failure leaves N appointments committed and M failed with no rollback. Consider wrapping in a transaction or documenting this as intentional per-item semantics.
5. **Frontend hardcodes terminal and company in BulkUpload** — `terminalId: "TRAPAC-LA"` and `truckingCompanyId: "default-company"` will fail against a real backend unless those IDs exist.
6. **Stubs never fail, return values ignored** — External service stubs always return success, and the service code never checks the `available` or `valid` flags. When real services are integrated, failures after the DB write will leave inconsistent state.
7. **No request body size limit** — The bulk endpoint accepts up to 50 appointments per request, each with unbounded transactions arrays. Add express body-size limits and consider capping transaction count per appointment.
8. **Frontend-backend validation mismatch** — The frontend doesn't show container number fields for DROP_EMPTY and PICK_EMPTY, but the backend requires them. Users will hit a 400 they can't fix from the UI.

## Auth posture: absent

The `authMiddleware` in `backend/src/middleware/auth.ts` reads `x-user-id` from the request header and assigns it to `req.userId`. If the header is missing, it defaults to `"system"`. The middleware never returns 401.

```typescript
export function authMiddleware(req, _res, next): void {
  const userId = req.headers["x-user-id"];
  req.userId = typeof userId === "string" ? userId : "system";
  next();
}
```

Any HTTP client can impersonate any user by setting the header. The audit trail records whatever user ID the caller provides, so it cannot be trusted for compliance purposes. There is no multi-tenancy enforcement; a trucking company dispatcher can view or cancel appointments belonging to a different company.

The frontend in `frontend/src/lib/api.ts` hardcodes `"x-user-id": "web-user"` on every request, so all frontend-originated actions appear as one user in the audit log.

## Appointment number generation and collision risk

`generateAppointmentNumber()` produces `APT-YYYYMMDD-XXXX` where `XXXX` is 4 random hex digits from `Math.random()`. This gives 65,536 unique values per calendar day. Birthday-paradox collisions become likely at a few hundred appointments per day.

The schema marks `appointmentNumber` as `@unique`, so a collision throws a Prisma unique constraint error that propagates as a 500 to the caller. There is no retry-on-collision logic.

```typescript
private generateAppointmentNumber(): string {
  const hex = Math.floor(Math.random() * 0xffff)
    .toString(16).padStart(4, "0").toUpperCase();
  return `APT-${dateStr}-${hex}`;
}
```

## Status lifecycle gap on the update path

The cancel endpoint enforces a strict allow-list: appointments with status CANCELLED, COMPLETED, CHECKED_IN, or NO_SHOW cannot be cancelled. The update endpoint enforces a similar check for CANCELLED, COMPLETED, and NO_SHOW — but then accepts *any string* as the new `appointmentStatus` value.

The Zod schema for update defines `appointmentStatus: z.string().optional()`, not an enum. A caller can PUT `{"appointmentStatus": "COMPLETED"}` on a PENDING appointment, skipping the check-in step. Or reverse a cancellation by first reaching an updatable state through another vector.

The service needs a state machine that defines legal transitions and rejects anything else.

## Bulk creation semantics

`createBulkAppointments` iterates sequentially and calls `createAppointment` for each item. Failures are captured per-item:

- A failure at item 3 of 10 leaves items 1-2 committed with no rollback.
- Each creation is its own Prisma operation with no transactional atomicity across the batch.
- The sequential loop means throughput scales linearly with batch size.

If the business requirement is all-or-nothing, this needs a transaction wrapper. If partial success is intentional, it should be documented explicitly.

## External service stubs: return values never checked

The service calls validation and slot services but never inspects their return values:

```typescript
// In createAppointment:
await this.validationService.validateReferences(...); // return value ignored
await this.slotService.checkAvailability(...);       // return value ignored
```

`checkAvailability` returns `{ available: boolean }` but the service never gates creation on `available === true`. `validateReferences` returns `{ valid: boolean, errors: [] }` but the service doesn't check `valid`. When real services are integrated, returning `available: false` or `valid: false` will still result in appointment creation.

The TOS and notification calls happen *after* the DB write. If they fail in production, the appointment is persisted but the downstream system never learns about it. There's no compensation pattern, no outbox, no retry queue.

## Frontend-backend validation mismatch

The frontend form's `showContainerNumber` condition shows the container number field only for PICK_IMPORT and DROP_EXPORT:

```typescript
// Frontend
const showContainerNumber =
  effectiveType === TransactionType.PICK_IMPORT ||
  effectiveType === TransactionType.DROP_EXPORT;

// Backend
const needsContainer = ["PICK_IMPORT", "DROP_EXPORT", "DROP_EMPTY", "PICK_EMPTY"];
```

A user creating a DROP_EMPTY or PICK_EMPTY appointment through the form will never see the container number field and will receive a 400 from the backend with no way to fix it from the UI.

## Frontend-backend contract mismatch in BulkUpload

The `BulkUpload` component hardcodes `terminalId: "TRAPAC-LA"` and `truckingCompanyId: "default-company"`. The seed data uses `"tc-test-001"` as the trucking company ID and `"term-la-001"` as the terminal ID. The terminal code `"TRAPAC-LA"` is used for the `code` field, not the `terminalId` field that the foreign key references. Both values will likely cause foreign key violations.

## Test coverage gaps

Not tested: concurrent appointment creation (collision handling), the slot-full / validation-failed paths (stubs always succeed), request body size limits, malformed date strings in time fields, the `containerNumber` filter path on `getAppointments`, date range filter interaction with timezone handling, and no integration test verifying the frontend-backend contract end-to-end.
